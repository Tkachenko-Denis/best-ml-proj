# Model Verification Checklist

Этот документ описывает чек-лист верификации для будущих релизов модели.
Перед тем, как выкатить новую версию, необходимо убедиться, что все пункты
из разделов ниже выполнены.

## 1. Метрики качества

- [ ] Accuracy на тестовом сете **> 0.85** (или другое целевое значение).
- [ ] F1-score по основному классу не ниже значения, достигнутого в предыдущем релизе.
- [ ] Проверка на переобучение: разница между train и test метриками не превышает заданного порога (например, 5 п.п.).
- [ ] Сравнение с предыдущей версией модели: новая версия не должна ухудшать ключевые метрики без объяснения причин.

## 2. Производительность

- [ ] Время предсказания на **1000 объектов < 200 ms** в целевой среде (например, локально или в docker-контейнере).
- [ ] Время обучения модели находится в допустимых рамках (зафиксировать ориентир, например, < 5 минут на текущем объёме данных).
- [ ] Пиковое потребление памяти при инференсе не превышает установленный лимит.
- [ ] При нагрузочном тестировании (batch-предсказание) модель сохраняет стабильное время отклика.

## 3. Проверка данных

- [ ] Проверка схемы входных данных: все обязательные признаки присутствуют, типы колонок соответствуют ожиданиям.
- [ ] Мониторинг дрифта данных по ключевым признакам (например, признак `feature_0`):  
  - [ ] Рассчитаны статистики (среднее, дисперсия, распределение) и сравнение с train-распределением.  
  - [ ] При обнаружении значимого дрифта (по KS-тесту, PSI и т. д.) — принято решение: обновление модели / алерт / комментарий.
- [ ] Обработка пропусков: убедиться, что все `NaN` и отсутствующие значения корректно обрабатываются пайплайном.
- [ ] Проверка категориальных признаков: все новые категории либо корректно мапятся, либо обработаны как “unknown”.

## 4. Безопасность и стабильность

- [ ] Модель корректно обрабатывает пропущенные значения и экстремальные значения (outliers) без выбросов ошибок.
- [ ] Добавлены и проходят тесты на “плохие” входные данные (пустой массив, слишком большие значения, неверные типы).
- [ ] Логи инференса не содержат ошибок и критических предупреждений.
- [ ] При нештатных ситуациях (например, ошибка при загрузке модели или некорректный формат входных данных) система возвращает понятное сообщение об ошибке и/или дефолтное поведение, а не падает.

## 5. Дополнительно (опционально)

- [ ] Обновлена документация (README, CHANGELOG) под новую версию.
- [ ] Зафиксированы версии зависимостей (requirements/pyproject).
- [ ] При необходимости обновлены скрипты деплоя / CI-конвейер.
